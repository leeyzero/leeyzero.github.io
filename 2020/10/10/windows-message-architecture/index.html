[иҜ‘] Windowsж¶ҲжҒҜдҪ“зі»

Posted on 2020-10-10 Edited on 2022-11-25 In ж“ҚдҪңзі»з»ҹ

еҺҹж–ҮеҮәиҮӘпјҡWindows Messaging Architecture

Most of the fresh software engineers and .NET developers may not know what really happens inside the Windows operating system when they implement and run a Windows application. This article explains the Windows Messaging Architecture of the Windows operating system.

еҫҲеӨҡж–°жүӢиҪҜд»¶е·ҘзЁӢеёҲе’Ң.NETејҖеҸ‘иҖ…еңЁзј–еҶҷе№¶иҝҗиЎҢдёҖдёӘWindowsеә”з”Ёж—¶пјҢд»–д»¬е№¶дёҚжё…жҘҡе…¶еңЁж“ҚдҪңзі»з»ҹдёӯзҡ„иҝҗдҪңеҺҹзҗҶгҖӮиҝҷзҜҮж–Үд»¶е°Ҷйҳҗиҝ°Windowsж“ҚдҪңзі»з»ҹзҡ„ж¶ҲжҒҜдҪ“зі»гҖӮ

зӘ—еҸЈе’ҢзӘ—еҸЈзұ»(Window and Window Class)

The Windows UI application (e) has one main thread (g), one or more > windows (a) and one or more child threads (k) [worker threads or UI threads].

Windowsеә”з”ЁзЁӢеәҸ(e)еҢ…еҗ«дёҖдёӘдё»зәҝзЁӢ(g)гҖҒеӨҡдёӘзӘ—еҸЈ(a)е’ҢдёҖдёӘжҲ–еӨҡдёӘеӯҗзәҝзЁӢ(k)[е·ҘдҪңзәҝзЁӢжҲ–UIзәҝзЁӢ]гҖӮ

An application must specify a window class and register with Windows (d) before it creates a window (a) and displays. A window class is a structure which contains the attributes of a window such as window styles, Icon, Cursor, Background color, menu resource name and window class name etc. Registering a window class associates a window procedure, class styles, and other class attributes with a class name. Each window class has an associated window procedure (c) shared by all windows (a) of the same class in an application. The window procedure processes messages for all windows of that class.

еңЁеҲӣе»әе№¶еұ•зӨәдёҖдёӘзӘ—еҸЈ(a)д№ӢеүҚпјҢеә”з”ЁзЁӢеәҸеҝ…йЎ»жҢҮе®ҡдёҖдёӘзӘ—еҸЈзұ»е№¶е°Ҷе…¶жіЁеҶҢеҲ°Windowsзі»з»ҹдёӯ(d)гҖӮзӘ—еҸЈзұ»еҸӘжҳҜдёҖдёӘж•°жҚ®з»“жһ„пјҢеҢ…еҗ«зӘ—еҸЈзҡ„дёҖдәӣеұһжҖ§пјҢеҰӮзӘ—еҸЈйЈҺж јгҖҒIconгҖҒCursorгҖҒзӘ—еҸЈиғҢжҷҜйўңиүІгҖҒиҸңеҚ•иө„жәҗеҗҚе’ҢзӘ—еҸЈзұ»еҗҚзӯүгҖӮжіЁеҶҢзҡ„зӘ—еҸЈзұ»йңҖиҰҒе…іиҒ”дёҖдёӘзӘ—еҸЈиҝҮзЁӢгҖҒзұ»йЈҺж јгҖҒе…¶е®ғзұ»еұһжҖ§е’ҢдёҖдёӘзӘ—еҸЈзұ»еҗҚгҖӮеңЁеә”з”ЁзЁӢеәҸдёӯпјҢе’ҢжҜҸдёӘзӘ—еҸЈзұ»е…іиҒ”е’ҢзӘ—еҸЈиҝҮзЁӢ(c)жҳҜжүҖжңүжӢҘжңүзӣёеҗҢзӘ—еҸЈзұ»еҗҚзҡ„зӘ—еҸЈ(a)е…ұдә«зҡ„гҖӮзӘ—еҸЈиҝҮзЁӢеӨ„зҗҶжүҖжңүжӢҘжңүзӣёеҗҢзӘ—еҸЈзұ»еҗҚзӘ—еҸЈзҡ„зӘ—еҸЈж¶ҲжҒҜгҖӮ

зӘ—еҸЈиҝҮзЁӢ(Window Procedure)

A window procedure (c) is a function that receives and processes all messages sent to the window (a). Every window class (b) has a window procedure (c), and every window (a) created with that class uses that same window procedure to respond to messages. The system sends a message to a window procedure by passing the message data as arguments to the procedure. The window procedure then performs an appropriate action for the message; it checks the message identifier and, while processing the message, uses the information specified by the message parameters.

зӘ—еҸЈиҝҮзЁӢ(c)жҳҜдёҖдёӘеҮҪж•°пјҢз”ЁдәҺжҺҘж”¶жүҖжңүеҸ‘еҫҖиҜҘзӘ—еҸЈ(a)зҡ„ж¶ҲжҒҜгҖӮжҜҸдёӘзӘ—еҸЈзұ»(b)йғҪжңүдёҖдёӘзӘ—еҸЈиҝҮзЁӢ(c)пјҢжҜҸдёӘз”ЁиҜҘзӘ—еҸЈзұ»еҲӣе»әзҡ„зӘ—еҸЈ(a)дҪҝз”ЁзӣёеҗҢзҡ„зӘ—еҸЈиҝҮзЁӢе“Қеә”ж¶ҲжҒҜгҖӮзі»з»ҹз»ҷзӘ—еҸЈиҝҮзЁӢеҸ‘йҖҒж¶ҲжҒҜжҳҜе°Ҷж¶ҲжҒҜдҪңдёәдёҖдәӣеҮҪж•°еҸӮж•°дј йҖ’з»ҷзӘ—еҸЈиҝҮзЁӢпјҢз„¶еҗҺзӘ—еҸЈиҝҮзЁӢеҶҚеҜ№е…·дҪ“зҡ„ж¶ҲжҒҜеҒҡе“Қеә”пјӣзӘ—еҸЈиҝҮзЁӢдјҡжЈҖжҹҘж¶ҲжҒҜж ҮиҜҶпјҢе№¶йҖҡиҝҮж¶ҲжҒҜж ҮиҜҶдҪҝз”Ёи·ҹиҜҘж¶ҲжҒҜзӣёеҜ№еә”зҡ„ж¶ҲжҒҜеҸӮж•°гҖӮ

A window procedure does not usually ignore a message. If it does not process a message, it must send the message back to the system for default processing. The window procedure does this by calling the DefWindowProc function (f), which performs a default action and returns a message result. The default window procedure function, DefWindowProc defines certain fundamental behavior shared by all windows. The default window procedure provides the minimal functionality for a window.

зӘ—еҸЈиҝҮзЁӢйҖҡеёёдёҚеҝҪз•Ҙж¶ҲжҒҜгҖӮеҰӮжһңеә”з”ЁдёҚеӨ„зҗҶжҹҗдёӘж¶ҲжҒҜпјҢеә”з”Ёеҝ…йЎ»е°ҶиҜҘж¶ҲжҒҜеҸ‘йҖҒеӣһзі»з»ҹпјҢдәӨз”ұзі»з»ҹеӨ„зҗҶгҖӮзӘ—еҸЈиҝҮзЁӢйҖҡиҝҮдёҖдёӘеҸ«DefWindowPrcеҮҪж•°жқҘеӨ„зҗҶиҝҷдәӣеҸ‘йҖҒеӣһзі»з»ҹзҡ„ж¶ҲжҒҜпјҢиҜҘеҮҪж•°дјҡдҪңй»ҳи®Өзҡ„е“Қеә”е№¶иҝ”еӣһж¶ҲжҒҜеӨ„зҗҶз»“жһңгҖӮDefWindowProcе®ҡд№үдәҶдёҖдәӣйҖӮз”ЁдәҺжүҖжңүзӘ—еҸЈзҡ„еҹәжң¬зҡ„иЎҢдёәпјҢе№¶жҸҗдҫӣдәҶдёҖдёӘзӘ—еҸЈзҡ„жңҖе°ҸеҠҹиғҪйӣҶгҖӮ

зӘ—еҸЈж¶ҲжҒҜ(Windows Message)

A Windows message (m) is nothing but a set of parameters such as a window handle, a message identifier and two values called message parameters (WPARAM and LPARAM). The window handle identifies the window for which the message is intended. The message identifier is a constant that identifies the purpose of a message. For example, the message identifier WM_PAINT tells the window procedure that the windowвҖҷs client area has changed and must be repainted.

зӘ—еҸЈж¶ҲжҒҜ(m)еҸӘжҳҜдёҖдёӘеҸӮж•°йӣҶеҗҲпјҢеҢ…жӢ¬зӘ—еҸЈеҸҘжҹ„гҖҒж¶ҲжҒҜж ҮиҜҶе’ҢдёӨдёӘиў«з§°дёәж¶ҲжҒҜеҸӮж•°(WPARAMе’ҢLPARAM)зҡ„еҖјгҖӮзӘ—еҸЈеҸҘжҹ„ж ҮиҜҶдёҖдёӘж¶ҲжҒҜеҚіе°ҶеҸ‘еҫҖзҡ„зӘ—еҸЈгҖӮж¶ҲжҒҜж ҮиҜҶжҳҜдёҖдёӘеёёйҮҸпјҢз”ЁдәҺи®ӨеҲ«дёҖдёӘж¶ҲжҒҜзҡ„е…·дҪ“еҗ«д№үгҖӮдҫӢеҰӮпјҢеҜ№дәҺж¶ҲжҒҜWM_PAINTз”ЁдәҺе‘ҠиҜүзӘ—еҸЈиҝҮзЁӢзӘ—еҸЈзҡ„е®ўжҲ·еҢәеҸ‘з”ҹдәҶж”№еҸҳйңҖиҰҒйҮҚж–°з»ҳеҲ¶гҖӮ

There are two types of Windows messages such as System defined messages and Application defined messages. The system sends or posts a system-defined message when it communicates with an application. An application can also send or post system-defined messages. For example, the WM_PAINT message identifier is used in system defined message which requests that a window paint its contents.

жңүдёӨз§Қзұ»еһӢзҡ„ж¶ҲжҒҜпјҢеҲҶеҲ«жҳҜзі»з»ҹе®ҡд№үж¶ҲжҒҜе’Ңеә”з”Ёе®ҡд№үж¶ҲжҒҜгҖӮеҪ“е’Ңеә”з”ЁзЁӢеәҸйҖҡдҝЎж—¶пјҢзі»з»ҹйҖҡиҝҮеҸ‘йҖҒ(sends)жҲ–жҠ•йҖ’(posts)дёҖдёӘзі»з»ҹе®ҡд№үж¶ҲжҒҜз»ҷеә”з”ЁзЁӢеәҸгҖӮдёҖдёӘеә”з”ЁзЁӢеәҸд№ҹеҸҜд»ҘеҸ‘йҖҒжҲ–жҠ•йҖ’зі»з»ҹж¶ҲжҒҜгҖӮдҫӢеҰӮпјҢWM_PAINTж¶ҲжҒҜжҳҜдёҖдёӘзі»з»ҹе®ҡд№үж¶ҲжҒҜпјҢиҜҘж¶ҲжҒҜиҜ·жұӮдёҖдёӘзӘ—еҸЈз»ҳеҲ¶е…¶зӘ—еҸЈеҶ…е®№гҖӮ

An application can create messages to be used by its own windows or to communicate with windows in other processes. If an application creates its own messages, the window procedure that receives them must interpret the messages and provide appropriate processing.

еә”з”ЁзЁӢеәҸеҸҜд»ҘеҲӣе»әдёәиҮӘиә«дҪҝз”ЁжҲ–з”ЁдәҺе’Ңе…¶е®ғиҝӣзЁӢйҖҡдҝЎзҡ„ж¶ҲжҒҜгҖӮеҰӮжһңдёҖдёӘеә”з”ЁзЁӢеәҸеҲӣе»әдёҖдёӘиҮӘе®ҡд№үзҡ„ж¶ҲжҒҜпјҢзӘ—еҸЈиҝҮзЁӢжҺҘж”¶еҲ°еҗҺеҝ…йЎ»и§Јжһҗе№¶йҖӮеҪ“еӨ„зҗҶиҜҘж¶ҲжҒҜгҖӮ

ж¶ҲжҒҜйҳҹеҲ—(Message Queues)

The Windows operating system uses two methods to route messages to a window procedure : posting messages to a first-in, first-out queue called a message queue (l), a system-defined memory (q) object that temporarily stores messages, and sending messages directly to a window procedure.

Windowsж“ҚдҪңзі»з»ҹдҪҝз”ЁдёӨдёӘж–№жі•жқҘеҸ‘йҖҒж¶ҲжҒҜз»ҷзӘ—еҸЈиҝҮзЁӢпјҡжҠ•йҖ’ж¶ҲжҒҜеҲ°дёҖдёӘе…Ҳиҝӣе…ҲеҮәзҡ„йҳҹеҲ—пјҢиҜҘйҳҹеҲ—иў«з§°дёәж¶ҲжҒҜйҳҹеҲ—(l)пјҢзі»з»ҹе®ҡд№үдәҶдёҖеқ—еҶ…еӯҳеҜ№иұЎ(q)дјҡдёҙж—¶еӯҳеӮЁзӣҙжҺҘеҸ‘еҫҖжҹҗдёӘзӘ—еҸЈиҝҮзЁӢзҡ„ж¶ҲжҒҜгҖӮ

The Windows operating system maintains only one System wide message queue (l) for the all applications. And, another message queue (j) is created for each graphical user interface (UI) thread. Whenever a user creates an UI thread, the message queue is also created for the UI thread.

Windowsж“ҚдҪңзі»з»ҹз»ҙжҠӨдәҶдёҖдёӘдҫӣжүҖжңүеә”з”ЁзЁӢеәҸжңҚеҠЎзҡ„зі»з»ҹзә§еҲ«зҡ„ж¶ҲжҒҜйҳҹеҲ—(l)гҖӮиҖҢеҸҰдёҖдёӘж¶ҲжҒҜйҳҹеҲ—(j)жҳҜиў«UIзәҝзЁӢеҲӣе»әзҡ„гҖӮеҪ“з”ЁжҲ·еҲӣе»әдёҖдёӘUIзәҝзЁӢзҡ„еҗҢж—¶пјҢзі»з»ҹдјҡдёәиҜҘUIзәҝзЁӢеҲӣе»әдёҖдёӘж¶ҲжҒҜйҳҹеҲ—гҖӮ

ж¶ҲжҒҜжөҒ(Message Flow)

DOS based applications make C-Runtime function calls to obtain input from the system. For example, it calls gets() C-Runtime function to get input from the keyboard. But, Windows based applications do not make explicit function calls to obtain input. Instead, they wait for the system to pass input to them. This is why Windows-based applications are event-driven.

еҹәдәҺDOSзҡ„еә”з”ЁзЁӢеәҸдҪҝз”ЁCиҝҗиЎҢж—¶еә“еҮҪж•°д»Һзі»з»ҹиҺ·еҸ–иҫ“е…ҘгҖӮдҫӢеҰӮпјҢеә”з”ЁзЁӢеәҸи°ғз”ЁCиҝҗиЎҢж—¶еә“еҮҪж•°gets()д»Һй”®зӣҳиҺ·еҸ–иҫ“е…ҘгҖӮдҪҶжҳҜпјҢеҹәдәҺзӘ—еҸЈзҡ„еә”з”ЁзЁӢеәҸе№¶дёҚйңҖиҰҒжҳҫзӨәзҡ„еҮҪж•°и°ғз”Ёд»ҘиҺ·еҸ–иҫ“е…ҘпјҢзӣёеҸҚпјҢеә”з”ЁзЁӢеәҸжҳҜзӯүеҫ…зі»з»ҹе°Ҷиҫ“е…Ҙдј йҖ’з»ҷд»–д»¬гҖӮиҝҷе°ұжҳҜдёәд»Җд№ҲеҹәдәҺзӘ—еҸЈзҡ„еә”з”ЁжҳҜдәӢд»¶й©ұеҠЁзҡ„гҖӮ

Whenever the user moves the mouse, clicks [1] the mouse buttons or types on the keyboard, the device driver (p) for the mouse or keyboard handovers [2] the details to User32.dll (n), which in turn converts the input into Windows messages and places[3] them in the system message queue (l).

з”ЁжҲ·ж— и®әжҳҜз§»еҠЁйј ж ҮгҖҒзӮ№еҮ»йј ж ҮжҢүй’®[1]жҲ–ж•ІеҮ»й”®зӣҳпјҢйј ж ҮжҲ–й”®зӣҳи®ҫеӨҮй©ұеҠЁ(p)дјҡе°ҶиҝҷдәӣеӨ„зҗҶдәӨз»ҷ[2]User32.dll(n)пјҢиҖҢUser32.dllеҸҲдјҡе°Ҷиҝҷдәӣиҫ“е…ҘиҪ¬еҢ–[3]жҲҗWindowsж¶ҲжҒҜе№¶е°Ҷиҝҷдәӣж¶ҲжҒҜеӯҳеҲ°зі»з»ҹж¶ҲжҒҜйҳҹеҲ—дёӯ(l)гҖӮ

The Windows operating system removes the messages, one at a time, from the system message queue, examines them to determine the destination window, and then posts [4] them to the message queue of the thread that created the destination window. A threadвҖҷs message queue receives all mouse and keyboard messages for the windows created by the thread. The thread removes [5] messages from its queue and directs the system to send [6 & 7] them to the appropriate window procedure for processing. How does a thread remove a message from threadвҖҷs message queue and process? LetвҖҷs discuss now.

ж“ҚдҪңзі»з»ҹеңЁжҹҗдёҖж—¶еҲ»д»Һзі»з»ҹж¶ҲжҒҜйҳҹеҲ—дёӯз§»йҷӨж¶ҲжҒҜпјҢжЈҖжҹҘе№¶еҶіе®ҡе…¶зӣ®ж ҮзӘ—еҸЈзЁӢеәҸпјҢз„¶еҗҺе°Ҷиҝҷдәӣж¶ҲжҒҜжҠ•йҖ’[4]з»ҷзӣ®ж ҮзӘ—еҸЈзЁӢеәҸеҲӣе»әзҡ„зәҝзЁӢж¶ҲжҒҜйҳҹеҲ— гҖӮзӘ—еҸЈеә”з”ЁзЁӢеәҸйҖҡиҝҮзәҝзЁӢеҲӣе»әзҡ„зәҝзЁӢж¶ҲжҒҜйҳҹеҲ—жҺҘж”¶жүҖжңүйј ж Үе’Ңй”®зӣҳж¶ҲжҒҜгҖӮиҜҘзәҝзЁӢд»Һж¶ҲжҒҜйҳҹеҲ—дёӯз§»йҷӨ[5]ж¶ҲжҒҜпјҢе№¶е°Ҷе…¶йҖҡиҝҮзі»з»ҹеҸ‘йҖҒ[6 & 7]з»ҷзӣёеә”зҡ„зӘ—еҸЈиҝҮзЁӢеӨ„зҗҶгҖӮзәҝзЁӢеҰӮдҪ•д»ҺзәҝзЁӢж¶ҲжҒҜйҳҹеҲ—дёӯз§»йҷӨдёҖжқЎж¶ҲжҒҜе‘ўпјҹжҲ‘д»¬й©¬дёҠе°ұдјҡи®Ёи®әгҖӮ

ж¶ҲжҒҜеҫӘзҺҜжҲ–ж¶ҲжҒҜжіө(Message Loop or Message Pump)

A Windows application has a WinMain() function which is the entry point for that application, like main()function in вҖҳCвҖҳ language program. The WinMain()function acts as applicationвҖҷs main thread function. The following three lines of code in the WinMain() function is referred as Message Loop or Message pump.

е°ұеғҸCиҜӯиЁҖзЁӢеәҸжңүдёҖдёӘmainеҮҪж•°дёҖж ·пјҢжҜҸдёҖдёӘWindowsеә”з”ЁйғҪжңүдёҖдёӘWinMainзҡ„е…ҘеҸЈеҮҪж•°гҖӮWinMainеҮҪж•°жү®жј”зқҖеә”з”ЁзЁӢеәҸдё»зәҝзЁӢеҮҪж•°и§’иүІгҖӮдёӢйқўиҝҷдёүиЎҢеңЁWinMainеҮҪж•°дёӯзҡ„д»Јз Ғиў«з§°дҪңж¶ҲжҒҜеҫӘзҺҜжҲ–ж¶ҲжҒҜжіөгҖӮ

The message loopвҖҷs GetMessage() call checks the message queue for windows messages. If it finds a valid message, it removes a message from its queue. After removing a message from its queue, an application can use the DispatchMessage() function to direct the system to send the message to a window procedure for processing. If there are no messages in the message queue, the GetMessage() call will be blocked till a valid message comes to the message queue.

ж¶ҲжҒҜеҫӘзҺҜдёӯзҡ„GetMessageи°ғз”ЁдјҡжЈҖжҹҘж¶ҲжҒҜйҳҹеҲ—дёӯзҡ„windowsж¶ҲжҒҜгҖӮеҰӮжһңеҸ‘зҺ°дёҖдёӘеҗҲжі•зҡ„ж¶ҲжҒҜпјҢиҜҘеҮҪж•°дјҡд»Һж¶ҲжҒҜйҳҹеҲ—дёӯз§»йҷӨдёҖжқЎж¶ҲжҒҜгҖӮеҪ“д»Һж¶ҲжҒҜйҳҹеҲ—дёӯз§»йҷӨдёҖжқЎж¶ҲжҒҜеҗҺпјҢеә”з”ЁзЁӢеәҸдҪҝз”ЁDispatchMessageеҮҪж•°е°ҶиҜҘж¶ҲжҒҜйҖҡиҝҮзі»з»ҹеҸ‘йҖҒз»ҷдёҖдёӘзӘ—еҸЈиҝҮзЁӢеӨ„зҗҶгҖӮеҰӮжһңж¶ҲжҒҜйҳҹеҲ—дёӯжІЎжңүж¶ҲжҒҜпјҢGetMessageи°ғз”Ёдјҡиў«йҳ»еЎһзӣҙеҲ°жңүдёҖдёӘеҗҲжі•зҡ„ж¶ҲжҒҜиҝӣе…Ҙж¶ҲжҒҜйҳҹеҲ—дёӯгҖӮ

In between these two calls, the TranslateMessage() method call does nothing, but simply translating virtual key messages into character messages. Your application still works, if you comment this line. Only thing is, your test team will file 50 keyboard related bugs.

еңЁGetMessageе’ҢDispatchMessageдёӯпјҢTranslateMessageеҮҪж•°и°ғз”ЁеҸӘжҳҜз®ҖеҚ•ең°еҒҡдәҶиҷҡжӢҹй”®ж¶ҲжҒҜе’Ңеӯ—з¬Ұж¶ҲжҒҜзҡ„иҪ¬жҚўгҖӮеҰӮжһңдҪ жіЁйҮҠиҜҘиЎҢд»Јз ҒпјҢдҪ зҡ„еә”з”ЁзЁӢеәҸд»Қз„¶иғҪеӨҹе·ҘдҪңпјҢеҸӘжҳҜдҪ зҡ„жөӢиҜ•е°Ҹз»„дјҡз»ҷжҠҘ50й”®зӣҳдәӢд»¶зӣёе…ізҡ„bugгҖӮ

ж—…иЎҢеҲ°жӯӨз»“жқҹ(Journey Ends Here)

The message loop (while loop) breaks when the GetMessage() retrieves a WM_QUIT message from the message queue and subsequently applicationвҖҷs main thread quits. ThatвҖҷs all, the applicationвҖҷs life comes to an end.

еҪ“GetMessageд»Һж¶ҲжҒҜйҳҹеҲ—дёӯжҺҘж”¶еҲ°WM_QUITж¶ҲжҒҜж—¶пјҢж¶ҲжҒҜеҫӘзҺҜ(whileеҫӘзҺҜ)дјҡдёӯжӯўпјҢйҡҸеҗҺеә”з”ЁзЁӢеәҸдё»зәҝзЁӢйҖҖеҮәгҖӮеӣ жӯӨпјҢеә”з”ЁзЁӢеәҸзҡ„ж•ҙдёӘз”ҹе‘Ҫе‘Ёжңҹе°ұз»“жқҹдәҶгҖӮ
